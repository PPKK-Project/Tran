
# Github 
저희가 깃허브로 작업할때 어떻게 해야하는지 단계별로 정리하고 주의사항 강조사항을 적어놨습니다.
모르겠는 부분은 얘기해주시면 설명 해드릴게요

## 📌 1단계: 작업 시작 및 환경 설정

1.  **메인 브랜치 이동:**
    ```bash
    git switch main
    ```
2.  **메인 브랜치 최신화:**
    ```bash
    git pull origin main
    ```
3.  **새 작업 브랜치 생성 및 이동:**
    ```bash
    git switch -c feature/기능이름
    ```
    (예: `feature/user-login`, `bugfix/cart-error`)

-----

## 💻 2단계: 작업 기록 (Add, Commit)

기능 개발 및 버그 수정 후, 변경 사항을 로컬 저장소에 저장합니다.

1.  **변경 상태 확인(해도되고 안해도 돼요):**
    ```bash
    git status
    ```
2.  **변경 파일 추가:**
    ```bash
    git add .
    ```
3.  **로컬 저장소에 커밋:**
    ```bash
    git commit -m "의미 있는 커밋 메시지"
    ```

-----

## 🚀 3단계: 원격 반영 및 통합 요청 (Push, PR)

로컬 작업을 원격 저장소에 반영하고 통합 과정을 시작합니다.

1.  **원격 저장소에 푸시:**
    ```bash
    # 최초 푸시 시:
    git push -u origin feature/기능이름
    # 이후 푸시 시:
    git push
    ```
2.  **Pull Request (PR) 생성:**
      * GitHub/GitLab 등 **웹사이트**에 접속합니다.
      * 푸시한 브랜치로 **`main` 브랜치를 향하는 PR**을 생성합니다.
      * PR 설명에 작업 내용을 상세히 기재하고, **리뷰어를 지정**합니다.

-----

## 🤝 4단계: 리뷰, 병합 및 정리

PR 승인 및 병합 후, 로컬 환경을 정리합니다.

1.  **코드 리뷰 반영:** 리뷰어의 피드백이 있으면 로컬에서 수정 후 2단계와 3단계(Commit, Push)를 반복합니다.
2.  **PR 병합 (웹사이트):** 리뷰어가 승인하면, PR을 \*\*`main` 브랜치로 병합(Merge)\*\*합니다.
3.  **로컬 메인 최신화:**
    ```bash
    git switch main
    git pull origin main
    ```
4.  **브랜치 정리:** (선택 사항이나 권장)
    ```bash
    # 로컬 브랜치 삭제
    git branch -d feature/기능이름
    # 원격 브랜치 삭제
    git push origin --delete feature/기능이름
    ```

-----

## 📢 강조 사항 및 주의 사항

### 🥇 강조 사항 (반드시 지켜야 할 원칙)

  * **메인 브랜치 직접 보호:** **`main`** 브랜치에는 절대 직접 커밋하거나 푸시하지 않습니다. 모든 코드는 PR을 통해서만 통합됩니다.
  * **PR 생성 의무:** 아무리 작은 변경이라도 코드를 통합하려면 반드시 PR을 열어 **최소 1명에게 리뷰**를 받아야 합니다.
  * **커밋 단위 명확화:** 커밋 하나에는 하나의 논리적인 변경(기능 추가 또는 버그 수정)만 담습니다. 커밋 메시지는 **제목과 본문**을 활용하여 명확하게 작성합니다.

### ⚠️ 주의 사항 (오류 및 문제 방지)

  * **작업 전 Pull:** 새 브랜치를 만들거나 작업을 시작하기 전에는 **항상** `git pull origin main`을 실행하여 최신 코드를 기반으로 시작합니다.
  * **민감 파일 관리:** API Key, 비밀번호 등 민감한 정보가 포함된 파일(예: `.env`, 설정 파일)은 **절대 커밋하지 않도록** \*\*`.gitignore`\*\*에 반드시 등록합니다.(저희는 이미 gitignore에 .env application.properties 를 추가 해두었는데 혹시 모르니 한번씩 확인 하시면서 하시면 될거같아요 한번 푸쉬되면 계속 추적되서 지우기가 힘들더라고요)
  * **충돌 해결은 로컬에서:** PR 병합 시 충돌(Conflict)이 발생하면, **로컬 작업 브랜치에서** `main` 브랜치를 Pull 받아 충돌을 해결한 후 다시 푸시해야 합니다.
  * **강제 푸시(`-f`) 금지:** \*\*`git push -f`\*\*는 원격 저장소의 기록을 덮어쓰므로, **기록 수정 등 불가피한 경우를 제외하고는** 사용을 금지합니다. 사용 시에는 반드시 팀원과 상의해야 합니다.
